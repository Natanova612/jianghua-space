### Phase 2：内容替换 + 真实化（预计 3-5 天，需用户配合提供内容）

目标：把所有 mock 内容替换为真实内容。

- [ ] 替换自传文案（用户已提供，直接复用）

- [ ] 接入真实照片（用户提供图片文件）

- [ ] 替换/确认 AI 应用案例

- [ ] 替换联系信息（用户提供邮箱/社交链接）
- [ ] 输出 Phase 2 日志

**阻塞项**：需要用户分批提供真实内容。建议在 Phase 2 开始前，先确认一份"内容提供清单"和排期。

### Phase 3：后端开发（预计 5-7 天，需确认技术方案）

目标：开发内容管理后台和星河 API。

**方案 A（轻量级）**：Decap CMS（基于 Git 的无头 CMS）
- 优点：无需数据库，内容变更自动提交到 Git，部署简单
- 缺点：姜华需要登录 CMS 编辑，星河也需要通过 CMS 更新，不能纯 API
- 适用：如果内容更新频率不高，且姜华接受 CMS 界面

**方案 B（中等复杂度）**：Astro API 路由 + Vercel KV / Upstash Redis
- 优点：可以支持 API 接口，星河可以程序化更新
- 缺点：需要数据库，部署稍复杂，需要写 API 鉴权
- 适用：如果星河需要自动/程序化更新内容

**方案 C（完整方案）**：Sanity/Strapi + Astro
- 优点：功能最完整，有完善的后台和 API
- 缺点：引入额外依赖，学习成本较高
- 适用：如果未来内容量很大，需要复杂的内容关系

**待用户确认**：选择哪种方案？
